# smithery 개념정리

---

>

## smithery 란? 

- Smithery는 **MCP 서버를 로컬에 설치하지 않고도** 원격 패키지에서 바로 “실행만” 시켜주는 런처
  - MCP 실행을 편하게 해주는 런너 일 뿐, 필수는 아님 
- **Smithery = 실행기(런처)**
- **npm = MCP 서버를 로컬에 설치하는 방식**

## 요약

- Smithery는 **“임시 다운로드 + 즉석 실행”** 방식이다
  - Smithery는 MCP 서버를 **로컬에 정식 설치하지 않고**, **실행할 때마다 원격에서 패키지를 가져와 실행**하도록 설계되어 있다.
  - 네트워크 없는경우 이용 불가능 
- 로컬에 영구 설치되지 않는다
- 실행은 로컬에서 한다
- 다만 실행할 때 네트워크로 패키지를 다운로드하여 로컬 임시 공간에 풀어서 실행한다

## 동작구조

1. MCP 서버 프로세스는 “로컬에서 동작”
   - 실제로 프로세스는 `/usr/bin/node …` 형태로 로컬에서 동작함
   - 클라이언트와의 IPC, socket, stdio 통신도 전부 **로컬 내부**
2. MCP 서버 바이너리는 “영구 설치되지 않음”
   - npm global 설치처럼 `/usr/local/bin/...` 에 저장되지 않음
   - 따라서 설치 목록에 나오지 않음
3. 실행할 때마다 최신 버전을 받아옴 (캐싱 가능)
   - 실행할 때 원격에서 패키지를 가져오는 구조라 **최신 버전 자동 적용**
   - 일부 캐시가 남을 수 있으나 정식 설치는 아님

## Smithery   VS   NPM 

1. **npm 설치 방식**

   → 프로그램을 PC에 “설치해두고” 언제든 실행하는 방식

2. **smithery 방식**

   → 프로그램을 설치하지는 않지만,  필요할 때마다 네트워크에서 가져와서 바로 실행하는 방식

| 항목           | **Smithery 사용**                                          | **NPM 직접 설치**                               |
| -------------- | ---------------------------------------------------------- | ----------------------------------------------- |
| 서버 설치 위치 | **로컬에 설치 안 함** (초경량 런처)                        | **로컬에 프로그램을 실제로 설치**               |
| 실행 방식      | 원격 레지스트리에서 MCP 서버 패키지를 **즉시 가져와 실행** | 설치된 바이너리/스크립트를 직접 실행            |
| 업데이트       | 항상 최신 버전 자동 사용                                   | `npm update -g` 해야 업데이트                   |
| 장점           | 설정 간단, 버전관리 자동, 설치 불필요                      | 로컬 독립 실행 가능, 빠르게 동작, 오프라인 가능 |
| 단점           | 인터넷 필요, 실행 속도 ↓                                   | 버전 관리 직접 해야 함, 로컬 환경 더러워짐      |
| 사용 목적      | **“빠르게 연결 테스트 / 최신 MCP를 바로 사용”**            | **“로컬 환경에 MCP 서버 고정 설치”**            |
| 보안           | 토큰/키를 args로 주입                                      | 동일하지만 로컬 파일로 관리 용이                |
